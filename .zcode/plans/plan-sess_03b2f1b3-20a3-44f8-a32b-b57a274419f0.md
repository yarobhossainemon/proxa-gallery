
# Hidden Albums Management Screen ‚Äî Implementation Plan

## Overview
Add a "Hidden Albums" entry in the Albums tab that navigates to a new dedicated screen for managing hidden albums. The screen reuses all existing `AlbumCustomization` infrastructure.

## Files to Modify

### 1. `AlbumCustomizationDao.kt` ‚Äî Add hidden-only query
- **Why**: Currently the DAO only has `getAllFlow()` which returns ALL customizations. We need a dedicated query for hidden-only albums to avoid filtering the full list in the ViewModel on every emission.
- **Change**: Add `@Query("SELECT * FROM album_customizations WHERE isHidden = 1") fun getHiddenFlow(): Flow<List<AlbumCustomization>>`

### 2. `AlbumCustomizationRepository.kt` ‚Äî Expose hidden albums Flow
- **Why**: The repository is the single point of access for data. The new screen needs a reactive Flow of only hidden album customizations.
- **Change**: Add `fun observeHidden(): Flow<List<AlbumCustomization>> = dao.getHiddenFlow()`

### 3. `GalleryUiState` (in `GalleryViewModel.kt`) ‚Äî Add `hiddenAlbums` field
- **Why**: The hidden albums list needs to be reactive and survive configuration changes. Adding it to the existing UiState keeps it consistent with the rest of the architecture.
- **Change**: Add `val hiddenAlbums: List<Album> = emptyList()` to `GalleryUiState`

### 4. `GalleryViewModel.kt` ‚Äî Expose hidden albums, unhide action, cover picker for hidden screen
- **Why**: All album actions flow through the ViewModel per MVVM architecture. We need:
  - `observeHiddenAlbums()`: Collects `albumCustomizationRepository.observeHidden()`, merges with `rawAlbums` to produce full `Album` objects (with cover URI, item count, display name), writes to `uiState.hiddenAlbums`. Runs at `init` alongside `observeAlbumCustomizations()`.
  - `unhideAlbum(albumId: Long)`: Sets `isHidden = false` via repository. The existing `observeAlbumCustomizations()` flow will re-emit, `applyCustomizationsToAlbums()` will re-include the album in the main list, and `observeHiddenAlbums()` will remove it from the hidden list. No restart needed.
  - `onEditHiddenAlbum(album: Album)`: Same as `onEditAlbum()` but used from the Hidden Albums screen ‚Äî opens the same `EditAlbumBottomSheet`.
- **Change**: Add these 3 methods + the observer. Reuse the same `EditAlbumBottomSheet` for rename/cover/reset from the hidden screen (it already has all needed actions).

### 5. `HomeScreen.kt` ‚Äî Add "Hidden Albums" entry + `onHiddenAlbumsClick` callback
- **Why**: The Albums tab needs a clickable section entry above the album grid.
- **Change**: 
  - Add `onHiddenAlbumsClick: () -> Unit = {}` parameter to `HomeScreen`
  - In the Albums tab (tab 2), after the "My Albums" header row and **before** `AlbumGridTab`, add a `SettingsItem` with `Icons.Rounded.VisibilityOff`, title "Hidden Albums", subtitle "View your hidden albums", `onClick = onHiddenAlbumsClick`
  - This is a **single entry**, not wrapped in a Card ‚Äî just a standalone `SettingsItem` row to match the navigation feel. Padding matches the Albums header (24.dp horizontal).
  - Only show the entry when there are hidden albums (`uiState.hiddenAlbums.isNotEmpty()`)

### 6. `GalleryNavHost.kt` ‚Äî Add route, wire navigation
- **Why**: Hidden Albums is a new screen requiring a navigation route.
- **Change**:
  - Add `private const val HIDDEN_ALBUMS_ROUTE = "hidden_albums"` route constant
  - Add `composable(HIDDEN_ALBUMS_ROUTE)` destination rendering `HiddenAlbumsScreen(...)`
  - Add `onHiddenAlbumsClick = { navController.navigate(HIDDEN_ALBUMS_ROUTE) }` to `HomeScreen` call
  - Add all necessary callbacks for the Hidden Albums screen actions (unhide, edit, cover picker, etc.)
  - Reuse the same `coverPickerLauncher` for cover changes from the hidden screen

### 7. NEW FILE: `HiddenAlbumsScreen.kt` ‚Äî The screen itself
- **Why**: A dedicated screen following the same pattern as `RecentlyDeletedScreen`.
- **Structure**:
  - `Scaffold` with a custom top bar (back arrow + "Hidden Albums" title)
  - Empty state using `GeminiEmptyState(icon = Icons.Rounded.VisibilityOff, title = "No Hidden Albums", subtitle = "Albums you hide will appear here.")`
  - Content: `LazyVerticalGrid(columns = GridCells.Fixed(2))` of `HiddenAlbumCard` composables
  - Each card displays: cover image (custom or latest), album name (custom or original), photo count, small "Hidden" badge
  - Card design reuses the same visual language as `AlbumCard` (same shape, overlay, press animation, shadows)
  - Overflow menu (MoreVert) on each card with: Unhide Album, Rename Album, Change Album Cover, Reset Customization
  - `EditAlbumBottomSheet` is reused for rename/cover/reset via the existing ViewModel methods
  - Unhide action uses `viewModel.unhideAlbum(albumId)` ‚Äî immediate removal from list

## Files NOT Modified
- `Album.kt` ‚Äî no changes needed, `Album` data class already has all fields
- `TrashDatabase.kt` ‚Äî no schema changes
- `AlbumCustomization.kt` ‚Äî entity unchanged
- Any theme files ‚Äî no new colors needed

## Reactive Data Flow
1. `observeHiddenAlbums()` collects `albumCustomizationRepository.observeHidden()` (Room Flow, `WHERE isHidden = 1`)
2. For each hidden customization, looks up the matching `Album` from `rawAlbums` (the MediaStore cache)
3. Merges customization fields onto the Album (customName, customCoverUri, etc.)
4. Validates cover URIs
5. Writes final `List<Album>` to `uiState.hiddenAlbums`
6. UI observes via `uiState.collectAsStateWithLifecycle()` ‚Äî automatic updates

When unhiding: Room write ‚Üí `observeHidden()` re-emits (album removed) ‚Üí `observeAlbumCustomizations()` re-emits ‚Üí `applyCustomizationsToAlbums()` re-includes album in main list. Both screens update immediately.
